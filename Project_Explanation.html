TrueLedger Architecture Guide

A breakdown of the project concept, blockchain simulation, and verification logic.

1. "Isn't it too simple?"

The student just uploads, admin approves. Is that it?

It looks simple on the surface, but that is the goal of good UX.

Under the hood, this is not just "saving a file." When an Admin clicks Approve, the system performs a State Transition that transforms a simple "image" into a Valid Digital Asset.

  • Before Approval: It is just an image file (jpg/png) with no trust value.
  • After Approval: It becomes a Cryptographically Signed Record. The system generates a unique verificationHash (a digital fingerprint).
The complexity isn't in the buttons; it's in the Trust Layer.

2. Where is the Blockchain?

Where is the blockchain involved here?

In this specific implementation (MVP/Demo version), we are using a **Cryptographic Ledger** approach which simulates the core value of blockchain: Immutability.

  • The "Block": System creates a `verificationHash` using SHA-256 (same as Bitcoin).
  • The "Chain": This hash is stored permanently.
  • The "Verification": If anyone tampers with the database, the Hash will no longer match.

Note: This local storage provides the proof of concept effectively for the demo.

3. Tamper-Proof Logic

How is it made tamper-free?
  1. Creation: Admin approves -> System calculates Hash `a1b2...`.
  2. Verification: User verifies -> System re-calculates Hash based on current current data.
  3. Detection: Checks `NewHash == OriginalHash`.

If a hacker changes "Grade B" to "Grade A" in the DB, the new hash will be `x9y8...`. The mismatch `x9y8... != a1b2...` triggers a TAMPER DETECTED alert.

4. Use Case Justification

Why will the student upload it if they already have it?

This solves the "Digitization of Legacy Records" problem.

  • Problem: Verifying old paper degrees takes weeks of phone calls.
  • Solution: Student validates it once via TrueLedger. Now they have a permanent digital link.
  • Benefit: Instant verification for employers.

5. Follow-Up Technical Q&A

How will the external party know it's signed by the institution?

The Verify Page proves it. The "Green Checkmark" confirms that the data mathematically matches the signature stored by the institution.


How will someone tamper? In what way?

Scenario: A database administrator hacks into the SQL database and manually updates a grade.

Detection: Without our hashing system, the website would show the fake grade. WITH our system, the verify tool recalculates the hash, sees the mismatch, and flags it as Invalid.

6. Technical Deep Dive (Show this Code)

Does the code actually compare the hash?

YES. Here is the logic from app/api/verify/route.ts:

// 🔐 RE-CALCULATE HASH to detect tampering
const currentHashData = `${certificate.id}:${certificate.fileUrl}:${certificate.createdAt.toISOString()}`;
const reCalculatedHash = crypto.createHash("sha256").update(currentHashData).digest("hex");

if (certificate.verificationHash !== reCalculatedHash) {
    return { message: "TAMPER DETECTED" };
}

This confirms that the system performs a real-time Cryptographic Integrity Check on every verification request.
